## Root Cause of the Data Freeze

**Two compounding defects, both silent.** The daily Action was scheduled at 08:00 UTC and triggered correctly every day. It failed at the very first step (checkout) before any of the actual work could run.

**Defect 1.** `data/sackmann/tennis_MatchChartingProject` was registered as a git submodule (mode 160000 in the index) but `.gitmodules` did not exist to declare its URL. The checkout step was configured `submodules: true`, so it tried to recursively initialize this entry and failed with `fatal: No url found for submodule path 'data/sackmann/tennis_MatchChartingProject' in .gitmodules`. The checkout exited with code 128. No subsequent steps ran. No error was visible on the site because the API kept serving old data successfully.

**Defect 2.** Even if checkout had worked, the daily action's "Run match scraper" step invoked `python3 -u scripts/scrape_atp_results.py` — a script that does not exist in the repo. It would have failed silently with `|| echo "Scraper failed"` swallowing the error.

Both defects existed since at least 2026-04-30 (oldest run in `gh run list`). For 8+ days nothing was actually being refreshed except whatever Robert ran manually.

## Honest List of Defects Not Fixed

1. **Glicko ratings frozen at 2026-03-25.** Per-player retirement is patched in-memory from supplement, but ratings, RDs, and peak_mu's reflect the March snapshot. Predictions for players who debuted in 2025 use defaults (1500 mu, high RD). Fixing requires the rules unlock to retrain Glicko.

2. **44 placeholder headshots silently degraded the cache.** They were 4729-byte uniform files served as legitimate-looking PNGs. The proxy's >500-byte threshold passed them but they rendered as blank/silhouette. Cleaned in this session, but if the original ingestion path is rerun naively it could re-introduce them.

3. **App Runner warm-cache loses state on every redeploy.** First request after deploy takes 30-60s for engine.load() to run and is not user-friendly. Out of scope here, but adding a /warmup endpoint that's pinged by the daily action could shave the first-user pain.

4. **Headshot scraper synthesizes codes that don't match canonical ATP IDs.** Future runs that need the real ATP code (e.g. for an ATP CDN integration if it ever opens up) won't be able to reuse our synthetic codes. Not load-bearing now but worth noting.

5. **Tablet (iPad portrait, 768px) navigation still shows soon-badge and full-width links** — the 767px breakpoint excludes 768px exactly. Not a horizontal-scroll defect; just looks slightly cramped. Could be tightened by changing breakpoint to `<=768`.
